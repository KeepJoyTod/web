# Figma → Vue 3（两步走）

## 目标

把用户提供的 Figma 链接转换为 Vue 3 前端代码，分两阶段完成：

1) **解析与方案**：解析figma链接的原型图并输出结构化蓝图（页面树、组件拆分、布局规则、样式/Token、交互）。
2) **生成代码（仅在用户确认后）**：按已确认的蓝图与仓库现有规范生成 Vue 3 SFC 及相关文件。

该技能用于避免“盲目代码生成”，通过先确认结构与样式决策来减少返工。

## 何时调用

当用户满足以下任一情况时调用：

- 提供 Figma URL（file/page/node 链接均可），并要求“转 Vue3 / 生成 Vue3 / 实现 UI”等
- 要求先“解析页面结构/组件/样式/布局”，确认后再生成代码

## 必要输入

- Figma 链接（file/page/node）以及目标页面/Frame（可指定多个）
- 约束条件（如有）：响应式断点、路由期望、组件边界、项目技术栈约束（如 Tailwind/纯 CSS 等）

若用户未提供约束，默认假设：

- 技术栈：Vue 3 + TypeScript + Pinia + Vite + Vue Router；HTTP 层采用 Axios 封装；组件库按仓库现状优先（Ant Design Vue 或 Element Plus）
- 基线体验：必须覆盖 loading/empty/error/disabled 等状态，提供骨架屏与占位图策略
- 状态管理：以 Pinia 管理用户、购物车、订单草稿、营销上下文；关键数据持久化（localStorage + 版本号）
- 质量门槛：不新增依赖（除非仓库已有或用户要求）；不输出注释（除非用户要求）

## PRD 原则（必须对齐）

在阶段 1 与阶段 2 的所有输出中，必须贯彻以下原则（来自前端 PRD）：

- 页面地图与导航：按“首页、类目、搜索、购物车、我的”建立路由与布局；子页面按业务链路组织（列表/详情/结算/支付结果/订单/售后/评价/个人中心/消息中心/登录注册）
- 关键交互：SKU 不可售置灰与原因提示；提交订单前置校验；提交中阻断二次点击；支付回跳后以最终状态展示，失败支持重试与更换方式
- 性能与体验：路由级按需加载、组件懒加载、图片懒加载；列表/详情数据短期缓存；关键链路预取（首页→列表→详情、下单前预取地址/默认发票）
- SEO/可索引（如适用）：商品详情页优先 SSR/预渲染能力；规范化 URL、面包屑、结构化数据（Product）；完善元信息与 Open Graph，处理 404/301
- 可访问性：语义化标签、键盘可达、对比度与焦点态、图片 alt，减少干扰动画
- 埋点与监控：至少覆盖曝光/点击、加购、结算、支付、评价与售后；补齐参数字典（商品/sku/来源位/活动上下文）；集成错误与性能监控（白屏/TTFB/FCP/LCP、路由时长、慢接口、JS/接口错误率）
- 接口契约：所有请求带 X-Client-Version；提交/创建类接口带 Idempotency-Key；按统一成功/错误结构处理错误码（含 RATE_LIMITED/UNAUTHORIZED/VALIDATION_FAILED 等），UI 必须给出可理解的提示与引导
- 安全与隐私：敏感数据不入日志；本地存储谨慎并可选加密；埋点具备隐私合规开关与匿名 ID 绑定策略

## 阶段 1：解析与蓝图（不改代码）

### 步骤 1：读取 Figma 链接

- 使用现有的 **Figma AI Bridge** 能力读取 Figma 链接内容。

### 步骤 2：输出蓝图

输出精简但完整的蓝图，包含：

- **页面/路由映射**：涉及的页面与主要屏幕（如适用），并标注所属全局导航与布局（主站/登录等）
- **组件树**：按层级拆分（page → section → reusable components）
- **布局规则**：grid/flex、间距、对齐、滚动区域、min/max 宽度、断点策略
- **样式 Token**：
  - 颜色、字号/行高、圆角、阴影、间距尺度
  - 图标/图片资源清单与使用位置
- **状态与交互**：hover/active/disabled/loading/empty/error；modal/drawer；导航交互；表单校验与防重复提交
- **数据占位**：哪些是静态/动态；必要时给出 props 的示例 JSON 结构；并标注接口契约要点（headers、幂等、错误码）
- **状态管理草图**：需要哪些 Pinia store、store 职责与持久化字段（含版本号）
- **埋点草图**：事件列表、触发点、参数字典、隐私开关点位
- **验收清单**：完成标准（像素对齐容忍度、响应式行为、基础无障碍、关键性能与可用性）

### 步骤 3：请求确认（硬门槛）

生成代码前，必须给出并让用户确认：

- 最终组件/文件规划（文件名 + 职责）
- 关键样式决策（Tailwind vs CSS Modules vs scoped CSS；Token 放哪里；图标策略）
- 路由与布局决策（导航归属、页面层级、守卫/登录态要求）
- 状态管理与持久化决策（Pinia store 划分、持久化键与版本策略）
- 交互关键点（提交阻断、错误/空态、可访问性焦点与键盘路径）
- 埋点清单与参数字典（至少覆盖 PRD 关键事件）
- 接口契约遵循方式（HTTP 封装、headers、幂等键、错误码映射到 UI）
- 不可避免的假设/风险点

并让用户明确选择其一：

- “按方案直接生成代码”
- “先调整方案（列出需要改的点）”

未获得明确确认前，禁止进入代码生成阶段。

## 阶段 2：生成 Vue 3 代码（用户确认后）

### 步骤 1：遵循仓库规范

写代码前必须先做：

- 检查现有 Vue 组件的命名与写法/样式约定
- 检查 `package.json` 的技术栈与样式库（Tailwind/UnoCSS/ElementPlus 等）
- 优先复用项目现有组件/工具方法，不重复造轮子

### 步骤 2：生成规则

- 使用 Vue 3 SFC，默认 `<script setup lang="ts">`
- 组件保持可组合与适度拆分，避免不必要的深层 props 传递
- 样式方案与仓库保持一致
- 除非仓库已使用，否则不新增依赖
- 除非用户明确要求，否则不写注释
- 页面必须实现 loading/empty/error/disabled/submit-in-progress 状态，并提供清晰文案与重试入口
- 接口调用必须通过统一 HTTP 封装：自动注入 X-Client-Version、X-Request-Id；提交类接口注入 Idempotency-Key；统一错误码到 UI 的映射
- 埋点必须通过统一入口上报：支持匿名 ID 与用户 ID 绑定；提供隐私合规开关
- 性能必须包含路由级懒加载与资源优化（图片懒加载、骨架屏、必要的预取点位）

### 步骤 3：验证

- 运行项目已有的 lint/typecheck/build 命令（如 `npm run build`/`npm run lint`/`npm run typecheck`）
- 确保页面可正常渲染且控制台无错误

## 输出格式（阶段 1）

输出建议包含：

- 组件树（列表）
- 文件规划表（文件 → 用途）
- Token 表（颜色/字体/间距等）
- Store/持久化规划（store → 职责 → 持久化字段）
- 埋点事件表（事件名 → 触发点 → 参数）
- 错误/空态清单（场景 → 提示文案 → 引导/重试）
- 需要用户确认的问题/假设（精简列表）

## 输出格式（阶段 2）

输出建议包含：

- 新增/修改文件清单（带链接）
- 本地运行/预览说明
- 与 Figma 的差异/限制说明（如字体不可用、图片资源缺失等）

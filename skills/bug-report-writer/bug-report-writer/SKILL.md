# Bug Report Writer

AI Bug提单助手 — 从问题发现到标准化Bug报告，一键完成。

## 技能概述

测试工程师每天约30%的时间花在Bug描述和沟通上。本技能旨在将Bug提单过程标准化、自动化：只需提供截图/日志/描述，AI自动生成结构清晰、信息完整的Bug报告。

### 核心优势

| 特性 | 说明 |
|------|------|
| 🖼️ **多模态输入** | 支持截图、错误日志、堆栈信息、自然语言描述 |
| 🔍 **智能分析** | 自动识别问题类型、提取关键错误、推断根因 |
| 📋 **标准化输出** | 生成符合团队规范的Bug单，信息完整不遗漏 |
| ⚡ **严重度定级** | 基于影响范围和用户受损程度，自动建议严重度/优先级 |
| 📤 **多平台适配** | 支持Jira、飞书、禅道、GitHub Issues等格式导出 |

## 使用场景

| 场景 | 关键词示例 |
|------|-----------|
| 截图提Bug | "这个页面有问题，帮我提个Bug"、"看这个截图，提个单" |
| 日志提Bug | "这个报错帮我提Bug"、"根据这个日志写Bug单" |
| 描述提Bug | "登录页面点登录没反应，帮我提Bug" |
| 批量提Bug | "这几个问题帮我一起提Bug" |
| Bug格式化 | "帮我把这个Bug描述规范一下"、"这个Bug单写得不清楚，优化下" |

## 支持的输入格式

### 🖼️ 截图输入

| 格式 | 扩展名 | 说明 |
|------|--------|------|
| APP截图 | .png, .jpg, .webp | 包含异常UI的截图 |
| PC截图 | .png, .jpg | Web端异常截图 |
| 录屏截图 | .gif | 操作过程截取 |

AI会从截图中识别：
- 异常UI元素（错位、缺失、遮挡、乱码）
- 错误弹窗/Toast信息
- 页面白屏/加载异常
- 数据展示错误

### 📝 文本输入

| 类型 | 说明 | 示例 |
|------|------|------|
| 错误日志 | 运行时错误输出 | `java.lang.NullPointerException at com.app.Login` |
| 堆栈信息 | Crash/Exception堆栈 | 完整的stack trace |
| 接口响应 | 异常的API响应 | `{"code": 500, "msg": "内部错误"}` |
| 自然语言 | 口头描述问题 | "点击提交按钮后页面卡住了" |
| 控制台日志 | 浏览器/ADB日志 | Chrome Console / Logcat输出 |

### 📎 组合输入

最佳实践：截图 + 日志 + 描述组合提供，AI能生成最完整的Bug单。

#### 禅道格式设计原则：研发导向

研发最关心的5件事，按优先级排列：
1. **标题** → 3秒判断是否自己负责的模块
2. **问题描述** → 一句话看懂全貌，不用读完整个步骤
3. **复现步骤** → 能否复现，验证问题真实存在
4. **错误信息** → 哪里报错，直接定位到接口/日志/错误码
5. **排查方向** → 从哪里查，减少盲目排查时间

## Bug分类体系

### 严重度定义

| 严重度 | 定义 | 占比建议 | 示例 |
|--------|------|----------|------|
| **S1-致命** | 系统崩溃、数据丢失、核心功能完全不可用 | <5% | 支付失败、登录崩溃、数据清空 |
| **S2-严重** | 核心功能受损但有替代方案，或重要数据错误 | 10-15% | 搜索无结果、订单状态错误 |
| **S3-一般** | 非核心功能异常，不影响主流程 | 40-50% | 分享失败、图片加载慢 |
| **S4-轻微** | UI瑕疵、文案错误、体验不佳 | 30-40% | 字体不一致、对齐偏移 |

### 优先级定义

| 优先级 | 定义 | 修复时限 | 示例 |
|--------|------|----------|------|
| **P0-紧急** | 线上事故，立即修复 | 2小时内 | 支付失败、服务宕机 |
| **P1-高** | 影响核心流程，本迭代修 | 24小时内 | 主流程阻断、数据错误 |
| **P2-中** | 影响非核心功能，排期修 | 本迭代内 | 辅助功能异常、体验问题 |
| **P3-低** | 体验优化，有空修 | 下迭代 | UI细节、文案修改 |

### Bug类型分类

| 类型 | 说明 | 示例 |
|------|------|------|
| **功能缺陷** | 功能不符合预期 | 按钮点击无响应、表单提交失败 |
| **UI缺陷** | 界面展示异常 | 布局错位、文字截断、样式不一致 |
| **性能问题** | 响应慢、卡顿、崩溃 | 页面加载>5s、内存泄漏 |
| **兼容性问题** | 特定设备/系统异常 | iOS正常Android异常、某机型闪退 |
| **数据异常** | 数据展示或存储错误 | 金额计算错误、列表重复 |
| **安全问题** | 权限绕过、信息泄露 | 越权访问、敏感信息明文 |
| **网络异常** | 弱网/断网处理不当 | 无网络提示缺失、请求超时无反馈 |

## 完整工作流程

### 第1步：接收问题输入

接收用户提供的问题信息，可以是：
- 截图文件（直接上传）
- 错误日志/堆栈信息（直接粘贴）
- 口头描述（自然语言）
- 组合输入（截图+日志+描述）

**收集信息清单**：
```
✅ 问题截图（如有）
✅ 错误日志/堆栈（如有）
✅ 问题描述（如有）
✅ 被测APP/系统名称
✅ 设备和环境信息（如用户未提供，AI给出默认值并提示补充）
```

### 第2步：分析问题

AI自动执行以下分析：

**2.1 问题识别**
- 从截图中识别异常类型（UI错位/白屏/报错弹窗/数据异常等）
- 从日志中提取关键错误信息（Exception/ErrorCode/Timeout等）
- 从描述中提取核心问题（什么操作→什么结果→什么预期）

**2.2 影响评估**
- 判断影响范围：单用户/部分用户/全部用户
- 判断是否阻塞主流程
- 判断是否有数据丢失风险
- 判断是否有安全风险

**2.3 严重度和优先级建议**
- 基于影响评估，自动建议S1-S4严重度
- 基于紧急程度，自动建议P0-P3优先级
- 提供建议理由

**2.4 根因推断**（辅助开发定位）
- 根据错误日志推断可能原因
- 根据问题现象关联已知Bug模式
- 提供排查方向建议

### 第3步：生成Bug报告

根据用户指定的或默认的输出格式生成Bug报告。

**Bug标题命名规范**：
```
格式：[模块] 问题简要描述

✅ 好的标题：
  - [登录] 点击登录按钮后页面无响应
  - [支付] 支付成功但订单状态未更新
  - [首页] 推荐列表在弱网下显示空白

❌ 差的标题：
  - 页面坏了
  - 有Bug
  - 登录问题
```

**复现步骤编写规范**：
1. 每步一个操作，不要合并步骤
2. 包含具体的操作对象（按钮名称、输入内容）
3. 最后一步描述观察到的现象
4. 步骤编号清晰，方便开发复现

**预期结果 vs 实际结果**：
- 预期结果：描述正确行为，不要写"不应该XX"
- 实际结果：描述错误行为，具体明确

### 第4步：格式化输出

根据目标平台格式化Bug报告：

| 目标平台 | 输出格式 | 说明 |
|----------|----------|------|
| Jira | Jira Wiki Markup | 支持Jira字段映射 |
| 飞书 | 飞书列表Markdown | 可粘贴到飞书文档 |
| 禅道 | Markdown表格 | 禅道Bug录入格式 |
| GitHub | GitHub Markdown | Issues格式 |
| 通用 | 标准Markdown | 默认格式 |

### 第5步：补充与确认

生成Bug报告后，提示用户补充缺失信息：

```
📋 Bug报告已生成！

⚠️ 建议补充以下信息：
- [ ] 具体的设备型号（当前为默认值）
- [ ] 发现问题的APP版本号
- [ ] 是否能稳定复现（必现/偶现）

是否需要调整严重度或优先级？
```

## 智能分析能力

### 从截图自动识别

| 识别能力 | 说明 |
|----------|------|
| 白屏/黑屏 | 页面未正常加载 |
| 错误弹窗 | Toast/Dialog/Alert中的错误信息 |
| UI错位 | 元素重叠、截断、超出屏幕 |
| 数据异常 | 数字错误、文字乱码、列表为空 |
| 加载状态 | 持续Loading、骨架屏不消失 |
| 崩溃页面 | ANR对话框、闪退截屏 |

### 从日志自动提取

| 提取能力 | 说明 |
|----------|------|
| Exception类型 | NullPointerException/TimeoutException等 |
| 错误码 | HTTP状态码、业务错误码 |
| 关键堆栈 | 最顶层的业务代码调用栈 |
| 时间线 | 请求发送→响应时间→异常时间 |
| 网络信息 | 请求URL、响应状态、耗时 |

### 根因推断模式

```
问题现象 → 日志分析 → 可能根因 → 排查建议

示例：
页面白屏 → 接口超时(SocketTimeoutException) → 后端服务不可用 → 检查服务健康状态
数据错误 → 返回code=200但data=null → 数据库查询异常 → 检查SQL和缓存
点击无响应 → 无Exception但UI线程阻塞 → 主线程执行耗时操作 → 检查是否在主线程做IO
```

## 注意事项

1. **截图质量**：确保截图清晰，包含完整的异常信息
2. **日志完整性**：尽量提供完整的错误堆栈，不要只截取部分
3. **复现步骤**：步骤越详细，开发定位越快
4. **严重度客观**：不要所有Bug都标S1，按实际影响定级
5. **敏感信息**：日志中可能包含Token/密码等，提交前需脱敏
6. **重复检查**：提单前建议搜索已有Bug，避免重复提单
7. **人工确认**：AI生成的Bug报告建议人工审核后再提交

```

```
📋 Bug报告已生成！

### 基本信息

### 实际结果

- [ ] 具体的设备型号和系统版本

📋 Bug报告已生成！

### 基本信息

### 错误信息
- 异常类型：SocketTimeoutException
- 发生位置：HttpClient.execute → OrderService.createOrder
- 推断原因：订单服务后端响应超时

### 复现步骤
1. 进入商品详情页
2. 点击「立即购买」
3. 确认订单信息，点击「提交订单」

1. 检查订单服务健康状态和响应时间

# Bug严重度定级详细指南

## 1. 定级总览

### 1.1 双维度评估模型

Bug严重度由两个维度共同决定：

```
严重度 = f(功能影响度, 用户受损度)

功能影响度：对系统功能的影响程度
用户受损度：对用户体验的损害程度

取两个维度中较高的级别作为最终严重度
```

### 1.2 严重度等级

| 等级 | 名称 | 颜色标识 | 核心判断标准 |
|------|------|----------|-------------|
| **S1** | 致命 | 🔴 红色 | 系统崩溃/数据丢失/核心功能完全不可用/安全漏洞 |
| **S2** | 严重 | 🟠 橙色 | 核心功能受损/重要数据错误/大面积用户受影响 |
| **S3** | 一般 | 🟡 黄色 | 非核心功能异常/影响部分用户/有替代方案 |
| **S4** | 轻微 | 🟢 绿色 | UI瑕疵/文案错误/体验不佳/极少用户受影响 |

---

## 2. S1-致命 详细标准

### 2.1 判定条件（满足任一即为S1）

| 条件类型 | 说明 | 示例 |
|----------|------|------|
| 系统崩溃 | APP闪退、ANR、服务宕机 | 支付时APP崩溃、首页白屏 |
| 数据丢失 | 用户数据永久性丢失 | 订单消失、收藏清空、消息丢失 |
| 核心功能完全不可用 | 主流程100%阻断 | 无法登录、无法下单、无法支付 |
| 安全漏洞 | 可被利用的安全缺陷 | 越权访问、SQL注入、敏感信息泄露 |
| 资金损失 | 直接导致用户或公司资金损失 | 支付金额计算错误、退款异常 |

### 2.2 S1场景速查表

| 场景 | 子场景 | 严重度 |
|------|--------|--------|
| 登录 | 所有用户无法登录 | S1 |
| 支付 | 支付流程中断 | S1 |
| 支付 | 金额计算错误 | S1 |
| 数据 | 数据库写入失败导致数据丢失 | S1 |
| 安全 | 用户Token可被窃取 | S1 |
| 崩溃 | 首页/核心页面100%闪退 | S1 |

### 2.3 S1 Bug处理要求

- **响应时间**：立即通知，15分钟内响应
- **修复时限**：2小时内出热修复
- **沟通要求**：即时通知开发负责人和产品
- **验证要求**：修复后必须在同环境验证

---

## 3. S2-严重 详细标准

### 3.1 判定条件（满足任一即为S2）

| 条件类型 | 说明 | 示例 |
|----------|------|------|
| 核心功能受损 | 核心功能异常但有替代方案 | 搜索无结果但可浏览分类 |
| 重要数据错误 | 关键数据显示或计算不正确 | 订单金额错误、库存显示不准 |
| 大面积用户受影响 | >30%用户受影响 | 特定系统版本大面积异常 |
| 高频功能异常 | 高频使用的功能不可用 | 刷新无响应、列表加载失败 |
| 性能严重下降 | 核心操作响应时间>10s | 首页加载超过10秒 |

### 3.2 S2场景速查表

| 场景 | 子场景 | 严重度 |
|------|--------|--------|
| 搜索 | 搜索结果为空（有关键词匹配商品） | S2 |
| 订单 | 订单状态更新延迟>5分钟 | S2 |
| 购物车 | 合计金额计算错误 | S2 |
| 消息 | 推送消息未送达 | S2 |
| 性能 | 核心页面加载>10s | S2 |
| 兼容性 | 主流机型（Top10）出现异常 | S2 |

### 3.3 S2 Bug处理要求

- **响应时间**：2小时内通知
- **修复时限**：24小时内
- **沟通要求**：当天同步给开发和产品
- **验证要求**：修复后需回归关联功能

---

## 4. S3-一般 详细标准

### 4.1 判定条件（满足任一即为S3）

| 条件类型 | 说明 | 示例 |
|----------|------|------|
| 非核心功能异常 | 辅助功能不可用 | 分享失败、评价提交失败 |
| 部分用户受影响 | <30%用户受影响 | 特定机型异常 |
| 有替代方案 | 核心功能有替代路径 | 搜索异常但可通过分类找到商品 |
| 偶发问题 | 出现概率<30% | 偶现的加载失败 |
| 性能一般下降 | 非核心操作响应慢 | 个人中心加载3-5秒 |

### 4.2 S3场景速查表

| 场景 | 子场景 | 严重度 |
|------|--------|--------|
| 分享 | 分享到微信失败 | S3 |
| 评价 | 提交评价后不显示 | S3 |
| 收藏 | 取消收藏后列表未更新 | S3 |
| 兼容性 | 非主流机型异常 | S3 |
| 性能 | 非核心页面加载3-5s | S3 |
| 偶发 | 10次操作出现1-2次异常 | S3 |

### 4.3 S3 Bug处理要求

- **响应时间**：当天记录
- **修复时限**：本迭代内
- **沟通要求**：纳入迭代排期
- **验证要求**：正常回归验证即可

---

## 5. S4-轻微 详细标准

### 5.1 判定条件（满足任一即为S4）

| 条件类型 | 说明 | 示例 |
|----------|------|------|
| UI瑕疵 | 不影响功能的视觉问题 | 字体大小不一致、图标偏移1-2px |
| 文案错误 | 文字描述有误但不影响理解 | 说明文案有错别字 |
| 体验不佳 | 不影响功能但体验不够好 | 动画不流畅、过渡不自然 |
| 极少用户受影响 | <5%用户且不影响核心流程 | 冷门机型特殊场景 |
| 建议优化 | 非Bug，改进建议 | 交互流程可以更顺畅 |

### 5.2 S4场景速查表

| 场景 | 子场景 | 严重度 |
|------|--------|--------|
| UI | 文字截断但信息完整 | S4 |
| UI | 图标间距不一致 | S4 |
| 文案 | 提示语有错别字 | S4 |
| 体验 | 列表滑动轻微卡顿 | S4 |
| 体验 | 动画播放不流畅 | S4 |

### 5.3 S4 Bug处理要求

- **响应时间**：有空时记录
- **修复时限**：下迭代或排期
- **沟通要求**：列入优化池
- **验证要求**：修复后简单确认即可

## 6. 严重度升级规则

### 6.1 自动升级条件

| 原始级别 | 升级条件 | 升级后级别 |
|----------|----------|-----------|
| S2 | 同一问题在24小时内被3+用户反馈 | → S1 |
| S3 | 同一问题影响>30%用户 | → S2 |
| S3 | 问题出现在核心流程中 | → S2 |
| S4 | 问题影响用户核心体验 | → S3 |

### 6.2 严重度争议处理

1. 测试和开发对严重度有分歧时，以**用户受损度**为主要判断依据
2. 如仍无法达成一致，由产品经理最终裁决
3. 安全类Bug一律不降级

---

## 7. 特殊场景定级指南

### 7.1 兼容性Bug

| 影响范围 | 严重度 |
|----------|--------|
| Top 10机型（覆盖>60%用户）出现异常 | S2 |
| Top 11-30机型出现异常 | S3 |
| 冷门机型出现异常 | S4 |
| 特定系统版本全量异常 | S1-S2（按影响用户量定） |

### 7.2 性能Bug

| 场景 | 响应时间 | 严重度 |
|------|----------|--------|
| 核心页面/操作 | >10s | S2 |
| 核心页面/操作 | 3-10s | S3 |
| 非核心页面/操作 | >10s | S3 |
| 非核心页面/操作 | 3-10s | S4 |
| 任何操作 | >30s | S2 |

### 7.3 间歇性Bug

| 出现概率 | 严重度调整 |
|----------|-----------|
| >70%（几乎必现） | 按正常标准定级 |
| 30%-70% | 降一级（最低S3） |
| <30% | 降一级（最低S4） |
| <10% | 最低S4，标注"偶现" |

> 注：S1级Bug不论出现概率如何，最低定为S2，不做降级

### 7.4 回归Bug

| 情况 | 严重度调整 |
|------|-----------|
| 修复后回归发现仍有问题 | 原严重度不变或升级 |
| 修复引入新Bug | 按新Bug单独定级 |
| 同一功能第3次以上回归 | 升一级，标注"反复出现" |

---

## 8. 优先级 vs 严重度

### 8.1 区别

| 维度 | 严重度(Severity) | 优先级(Priority) |
|------|------------------|------------------|
| 含义 | Bug本身的严重程度 | 修复的紧迫程度 |
| 决定者 | 测试（客观评估） | 产品/开发（业务决策） |
| 参考维度 | 功能影响、用户受损 | 上线计划、修复成本、业务紧急度 |

### 8.2 映射关系（默认）

| 严重度 | 默认优先级 | 可能调整的场景 |
|--------|-----------|---------------|
| S1 | P0 | 无，S1必为P0 |
| S2 | P1 | 如当前无发布计划→P2；如影响核心大客户→P0 |
| S3 | P2 | 如是关键客户反馈→P1；如资源紧张→P3 |
| S4 | P3 | 如是老板反馈→P1（😄） |

### 8.3 优先级异常场景

严重度高但优先级低的例子：
- S2 Bug，但只在即将下线的功能中出现 → P3
- S2 Bug，但只在内部测试环境出现，生产环境不受影响 → P2

严重度低但优先级高的例子：
- S4 文案错误，但涉及合规要求 → P1
- S4 UI瑕疵，但影响品牌发布会演示 → P1

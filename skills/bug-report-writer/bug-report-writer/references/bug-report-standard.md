# Bug报告编写标准

## 1. Bug报告核心原则

### 1.1 5W1H原则

每份Bug报告必须回答以下6个问题：

| 维度 | 问题 | 对应字段 |
|------|------|----------|
| **Who** | 谁发现了问题？什么角色受影响？ | 报告人、影响用户群 |
| **What** | 出了什么问题？ | Bug标题、实际结果 |
| **When** | 什么时候出现？ | 发现时间、触发条件 |
| **Where** | 在哪里出现？ | 所属模块、页面、环境 |
| **Why** | 为什么是Bug？ | 预期结果 vs 实际结果的差距 |
| **How** | 如何复现？ | 复现步骤 |

### 1.2 一Bug一问题

- ❌ 不要在一个Bug单中报告多个不相关问题
- ✅ 每个Bug单只报告一个具体问题
- ✅ 如果多个问题有因果关系，可以在"关联Bug"中引用

### 1.3 客观描述

- ❌ "这个功能太烂了"、"开发写的代码有Bug"
- ✅ "点击提交按钮后，页面无响应持续超过30秒"

## 2. Bug标题规范

### 2.1 命名格式

```
[模块名] 问题简要描述
```

### 2.2 模块名映射

| 模块 | 标准命名 | 示例 |
|------|----------|------|
| 登录/注册 | [登录] / [注册] | [登录] 验证码发送失败 |
| 首页 | [首页] | [首页] Banner图片不显示 |
| 搜索 | [搜索] | [搜索] 搜索结果为空无提示 |
| 商品详情 | [商品详情] | [商品详情] 规格选择弹窗闪退 |
| 购物车 | [购物车] | [购物车] 合计金额计算错误 |
| 订单 | [订单] | [订单] 取消订单后库存未恢复 |
| 支付 | [支付] | [支付] 微信支付回调丢失 |
| 个人中心 | [个人中心] | [个人中心] 头像上传后显示默认图 |
| 消息 | [消息] | [消息] 推送消息点击跳转空白页 |
| 设置 | [设置] | [设置] 退出登录后仍收到推送 |

### 2.3 标题质量标准

| 等级 | 标准 | 示例 |
|------|------|------|
| ✅ 优秀 | 模块+现象+条件 | [购物车] 弱网下修改数量后合计金额未更新 |
| ✅ 良好 | 模块+现象 | [购物车] 修改数量后合计金额未更新 |
| ⚠️ 一般 | 只有现象 | 合计金额没更新 |
| ❌ 差 | 模糊不清 | 购物车有问题 |

## 3. 复现步骤规范

### 3.1 编写原则

1. **可复现**：他人按步骤能复现问题
2. **具体化**：操作对象明确，不使用"某按钮"
3. **最小化**：去除不相关的步骤
4. **编号化**：每步有明确编号

### 3.2 步骤模板

```
1. [前置操作] 打开XXX，进入XXX页面
2. [核心操作] 点击/输入/滑动XXX
3. [触发操作] 执行XXX操作
4. [观察结果] 观察到XXX现象
```

### 3.3 不同类型Bug的步骤模板

#### 功能缺陷

```
1. 打开APP，进入[页面]
2. 执行[操作]
3. 观察[异常现象]
```

#### UI缺陷

```
1. 打开APP，进入[页面]
2. 使用[特定设备/分辨率]
3. 观察[UI异常]
```

#### 性能问题

```
1. 打开APP，进入[页面]
2. [操作前记录时间T1]
3. 执行[操作]
4. [操作完成记录时间T2]
5. 响应时间 = T2 - T1 = [X秒]，超过[阈值]
```

#### 崩溃/闪退

```
1. 打开APP，进入[页面]
2. 执行[操作A]
3. 执行[操作B]
4. APP崩溃/闪退，返回桌面
5. 重新打开APP，[描述恢复状态]
```

#### 间歇性Bug

```
1. 打开APP，进入[页面]
2. 执行[操作]
3. 问题间歇性出现（约X次/Y次，出现概率约Z%）
4. 出现时现象：[描述]
5. 未出现时行为：[描述]

备注：该问题在[特定条件]下更容易复现
```

## 4. 预期结果 vs 实际结果

### 4.1 编写原则

| 原则 | 预期结果 | 实际结果 |
|------|----------|----------|
| 描述正确行为 | ✅ 应该怎样 | ❌ 不要写"不应该怎样" |
| 具体明确 | ✅ "金额显示为¥99.00" | ❌ "金额显示不对" |
| 可验证 | ✅ "跳转到首页" | ❌ "页面正常" |
| 含数据 | ✅ "列表显示10条数据" | ❌ "列表有数据" |

### 4.2 示例对比

| 场景 | ❌ 差的写法 | ✅ 好的写法 |
|------|-----------|-----------|
| 登录失败 | 预期：登录正常 / 实际：登录有问题 | 预期：输入正确账号密码后登录成功，跳转首页 / 实际：点击登录后弹出"系统异常"提示 |
| 金额错误 | 预期：金额正确 / 实际：金额错误 | 预期：合计金额显示¥199.00（单价¥99.00 × 数量2 + 运费¥1.00）/ 实际：合计金额显示¥99.00（仅显示单价） |
| 白屏 | 预期：页面正常 / 实际：白屏 | 预期：页面加载完成后显示商品列表 / 实际：页面始终显示空白，Loading图标消失 |

## 5. 环境信息规范

### 5.1 必填环境字段

| 字段 | 说明 | 示例 |
|------|------|------|
| 操作系统 | 系统类型和版本 | Android 13 / iOS 16.5 / Windows 11 |
| 设备型号 | 具体设备 | Samsung Galaxy S23 / iPhone 15 Pro |
| APP版本 | 当前测试版本 | v2.1.0 (build 20260423) |
| 网络环境 | 网络类型 | WiFi / 4G / 5G / 弱网(2G) |

### 5.2 选填环境字段

| 字段 | 说明 | 适用场景 |
|------|------|----------|
| 屏幕分辨率 | 设备分辨率 | UI缺陷 |
| 浏览器版本 | Web测试 | Web端Bug |
| 服务端版本 | 后端版本号 | 接口相关Bug |
| 数据库版本 | DB版本 | 数据相关Bug |
| 第三方SDK版本 | 依赖SDK版本 | 支付/推送等集成问题 |

## 6. Bug生命周期

```
新建(New) → 确认(Confirmed) → 处理中(In Progress) → 已修复(Resolved) → 已验证(Closed)
    ↓              ↓                ↓                    ↓
  拒绝(Rejected)  重新打开(Reopened)   不修复(Won't Fix)    重新打开(Reopened)
```

### 状态流转说明

| 状态 | 含义 | 下一个状态 |
|------|------|-----------|
| **New** | 测试刚提交 | Confirmed / Rejected |
| **Confirmed** | 开发确认是Bug | In Progress |
| **In Progress** | 开发修复中 | Resolved |
| **Resolved** | 开发已修复，待验证 | Closed / Reopened |
| **Closed** | 测试验证通过 | Reopened（如回归发现仍有问题） |
| **Rejected** | 不是Bug/重复/不修复 | New（如有异议可重开） |
| **Reopened** | 验证未通过，重新打开 | In Progress |

## 7. Bug报告质量检查清单

提交前自查：

- [ ] 标题是否包含模块名和问题简述？
- [ ] 严重度和优先级是否合理？
- [ ] 复现步骤是否完整且可执行？
- [ ] 预期结果是否明确且可验证？
- [ ] 实际结果是否描述具体现象？
- [ ] 环境信息是否填写完整？
- [ ] 是否有截图/日志作为佐证？
- [ ] 是否搜索了重复Bug？
- [ ] 敏感信息是否已脱敏？
- [ ] 影响范围是否已评估？

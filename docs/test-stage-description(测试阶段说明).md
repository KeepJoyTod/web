# 测试阶段说明文档

本文档用于说明项目测试各阶段的目标、输入、执行内容、产出物和进入下一阶段的条件，适合作为测试执行说明。

## 1. 需求评审与范围确认

| 项目 | 说明 |
| --- | --- |
| 目标 | 明确本轮测试测什么、不测什么、通过标准是什么 |
| 输入 | PRD、接口文档、原型、开发排期 |
| 主要动作 | 识别核心业务链路、风险点、边界条件、性能目标 |
| 产出物 | 测试范围、风险清单、阶段目标 |
| 进入下一阶段条件 | 范围已确认，核心需求无明显歧义 |

说明：

- 这一阶段要先把公开链路、登录链路、下单链路、支付链路等关键业务拆出来。
- 如果性能测试是本轮目标，需要在这里提前确定并发、时长和阈值。

## 2. 测试计划

| 项目 | 说明 |
| --- | --- |
| 目标 | 安排测试顺序、资源、时间和责任人 |
| 输入 | 已确认的测试范围和排期 |
| 主要动作 | 确定测试类型、测试环境、执行顺序、通过标准 |
| 产出物 | 测试计划、测试排期、责任分工 |
| 进入下一阶段条件 | 各阶段安排可执行，依赖项已识别 |

说明：

- 建议先冒烟，再功能，再回归，再性能。
- 不建议在主流程大量失败时直接做压测，结果没有参考价值。

## 3. 测试设计

| 项目 | 说明 |
| --- | --- |
| 目标 | 把测试范围细化为可执行的测试项 |
| 输入 | 测试计划、功能需求、接口文档 |
| 主要动作 | 编写测试用例、设计异常场景、准备测试数据 |
| 产出物 | 测试用例文档、测试数据方案 |
| 进入下一阶段条件 | 关键流程均已覆盖 |

本项目已有测试用例文档示例：

- `docs/login-test-cases.md`
- `docs/register-test-cases.md`
- `docs/navigation-test-cases.md`
- `docs/product-detail-test-cases.md`
- `docs/cart-test-cases.md`
- `docs/checkout-test-cases.md`

## 4. 环境与数据准备

| 项目 | 说明 |
| --- | --- |
| 目标 | 保证测试环境可运行、可复现 |
| 输入 | 部署包、数据库脚本、配置文件、测试账号 |
| 主要动作 | 启动前后端、准备数据库、准备 Redis、准备种子数据、检查日志和监控 |
| 产出物 | 可用测试环境、测试账号、测试数据 |
| 进入下一阶段条件 | 核心服务可访问，关键依赖正常 |

说明：

- 自动化和压测都依赖稳定的环境与数据。
- 如果环境本身不稳定，后面拿到的失败结果大概率无法区分是系统缺陷还是环境问题。

## 5. 冒烟测试

| 项目 | 说明 |
| --- | --- |
| 目标 | 快速确认系统是否具备继续测试的基础 |
| 输入 | 已部署环境 |
| 主要动作 | 验证首页、登录、核心接口、数据库连接、关键页面跳转 |
| 产出物 | 冒烟测试结果 |
| 进入下一阶段条件 | 核心功能可打开、可请求、可返回 |

说明：

- 冒烟测试失败时，应优先停止后续测试并修复阻塞问题。
- 这是成本最低、收益最高的一道前置过滤。

## 6. 功能测试

| 项目 | 说明 |
| --- | --- |
| 目标 | 验证业务功能正确性 |
| 输入 | 测试用例、测试数据 |
| 主要动作 | 正常流程、异常流程、边界条件、权限控制、页面交互验证 |
| 产出物 | 功能测试记录、缺陷列表 |
| 进入下一阶段条件 | 核心业务链路基本稳定 |

本项目自动化资产主要在：

- `frontend/tests/login.spec.ts`
- `frontend/tests/register.spec.ts`
- `frontend/tests/navigation.spec.ts`
- `frontend/tests/product-detail.spec.ts`
- `frontend/tests/cart.spec.ts`
- `frontend/tests/checkout.spec.ts`

## 7. 接口 / 集成测试

| 项目 | 说明 |
| --- | --- |
| 目标 | 验证前后端、模块间、服务间联调正确 |
| 输入 | 接口文档、联调环境 |
| 主要动作 | 校验请求参数、返回结构、鉴权、状态码、异常分支 |
| 产出物 | 接口测试结果、联调问题清单 |
| 进入下一阶段条件 | 主要接口可稳定支撑主流程 |

说明：

- 这一阶段会暴露很多“功能看起来对，接口行为其实不一致”的问题。
- 例如公开资源是否被误拦截、登录态与游客态逻辑是否混用。

## 8. 回归测试

| 项目 | 说明 |
| --- | --- |
| 目标 | 验证缺陷修复后没有引入新问题 |
| 输入 | 已修复缺陷、回归范围 |
| 主要动作 | 重跑受影响模块与主流程自动化测试 |
| 产出物 | 回归结果、剩余风险说明 |
| 进入下一阶段条件 | 阻塞缺陷关闭，主流程回归通过 |

说明：

- 功能修复后必须回归，不能只看单点验证。
- 如果主流程自动化不稳定，不建议进入压测和验收。

## 9. 性能测试

| 项目 | 说明 |
| --- | --- |
| 目标 | 验证系统在目标负载下的性能和稳定性 |
| 输入 | 稳定版本、压测脚本、压测环境、性能阈值 |
| 主要动作 | 执行基准测试、负载测试、压力测试、稳定性测试 |
| 产出物 | 压测报告、指标数据、瓶颈分析 |
| 进入下一阶段条件 | 指标达标，或已明确瓶颈与优化计划 |

本项目已有压测资产：

- `k6/api-load.js`
- `scripts/run-k6-and-record.ps1`
- `docs/performance-test-records/`

重点说明：

- 压测结束后的下一步通常是“分析 -> 调优 -> 回归 -> 复测”。
- 如果 `http_req_failed`、`checks`、`p95`、`p99` 不达标，不能直接给出通过结论。

## 10. 缺陷管理与调优闭环

| 项目 | 说明 |
| --- | --- |
| 目标 | 让问题被记录、被修复、被验证，而不是停留在口头结论 |
| 输入 | 缺陷现象、日志、截图、压测指标 |
| 主要动作 | 提单、分级、定位、修复、验证、关闭 |
| 产出物 | 缺陷报告、修复记录、复测结论 |
| 进入下一阶段条件 | 阻塞问题关闭，剩余风险可接受 |

本项目建议统一沉淀到：

- `docs/bug-report.md`

说明：

- 压测发现的问题也应该按缺陷或优化项管理，而不是只写一句“性能不行”。
- 例如慢 SQL、接口 `500`、鉴权误拦截、测试脚本兼容性问题，都应有明确记录。

## 11. 验收测试 / UAT

| 项目 | 说明 |
| --- | --- |
| 目标 | 确认版本是否满足业务上线要求 |
| 输入 | 已通过内部测试的版本 |
| 主要动作 | 由产品、业务或验收方验证关键业务场景 |
| 产出物 | 验收结论、上线建议 |
| 进入下一阶段条件 | 验收通过 |

说明：

- 验收重点不是替代功能测试，而是确认业务目标是否达成。
- 如果验收阶段还在频繁发现基础功能问题，说明前置测试不够扎实。

## 12. 上线前检查与上线后观察

| 项目 | 说明 |
| --- | --- |
| 目标 | 降低发布风险，确认上线后运行稳定 |
| 输入 | 验收通过版本、发布方案、回滚方案 |
| 主要动作 | 核对配置、数据库脚本、监控告警、回滚预案，上线后观察日志和指标 |
| 产出物 | 发布检查记录、上线观察结论 |
| 进入下一阶段条件 | 发布完成且观察窗口内无严重异常 |

上线后建议关注：

- 错误率是否明显升高
- 核心接口响应时间是否恶化
- CPU、内存、数据库连接是否异常
- 订单、支付、购物车等核心链路是否有真实用户报错

## 13. 建议执行原则

1. 先保证冒烟和主流程稳定，再做性能测试。
2. 压测失败后先分析瓶颈，再调优和复测，不要直接跳结论。
3. 所有缺陷都要有记录、有验证结果、有关闭条件。
4. 测试结论必须基于产物，例如测试报告、自动化结果、压测记录，而不是口头判断。

## 助手 · 2026-04-27T11:29:46.380Z

按你这个项目，K6 压测建议主要压后端 API，不压 Vue 页面。页面跳转功能继续用 Playwright；K6 负责验证 `/api` 后端在并发下的响应时间、错误率、吞吐和数据库压力。

**压测范围**

基础地址：

```text
http://127.0.0.1:8080/api
```

优先压这些接口：

```text
GET  /api/
GET  /api/v1/products?page=1&size=10
GET  /api/v1/products/{id}
POST /api/v1/auth/login
GET  /api/v1/cart
GET  /api/v1/orders
```

下单、支付、售后这类写接口先不要大并发乱压，容易污染数据库。要压的话单独准备测试库和测试账号。

**执行方案**

1. 冒烟压测  
   1 个虚拟用户，跑 1 分钟，确认脚本、服务、账号、接口都通。

2. 基准压测  
   10 VU，跑 5 分钟，拿到正常状态下的平均响应时间、P95、错误率。

3. 阶梯压测  
   10 → 30 → 50 → 80 VU，每档 3 到 5 分钟，观察系统从正常到吃紧的拐点。

4. 峰值压测  
   30 秒内升到 100 或 150 VU，保持 1 到 2 分钟，再降下来，看突增流量是否会导致 5xx、超时、连接池耗尽。

5. 稳定性压测  
   30 到 50 VU，跑 30 到 60 分钟，看是否有内存、连接、Redis/MySQL 慢查询问题。

你的后端 Hikari 连接池最大是 `20`，所以本地压到 50+ VU 时如果响应变慢，不一定是接口逻辑坏了，可能是数据库连接池已经成为瓶颈。

**建议指标**

```text
http_req_failed < 1%
checks > 99%
GET 商品列表 P95 < 500ms
登录/购物车/订单 P95 < 800ms
P99 < 1500ms
无 5xx
无明显 MySQL 慢查询
CPU 持续不要超过 80%
```

**脚本骨架**

可以建 `k6/api-load.js`：

```js
import http from 'k6/http'
import { check, sleep } from 'k6'

const BASE_URL = __ENV.BASE_URL || 'http://127.0.0.1:8080/api'

export const options = {
  stages: [
    { duration: '1m', target: 10 },
    { duration: '3m', target: 30 },
    { duration: '3m', target: 50 },
    { duration: '1m', target: 0 },
  ],
  thresholds: {
    http_req_failed: ['rate<0.01'],
    http_req_duration: ['p(95)<800', 'p(99)<1500'],
    checks: ['rate>0.99'],
  },
}

export function setup() {
  const res = http.post(
    `${BASE_URL}/v1/auth/login`,
    JSON.stringify({
      account: __ENV.ACCOUNT || 'user@example.com',
      password: __ENV.PASSWORD || '123456',
    }),
    { headers: { 'Content-Type': 'application/json' } },
  )

  check(res, {
    'login success': r => r.status === 200 && Boolean(r.json('data.token')),
  })

  return { token: res.json('data.token') }
}

export default function (data) {
  const headers = {
    Accept: 'application/json',
    'Content-Type': 'application/json',
  }

  let res = http.get(`${BASE_URL}/`, { headers })
  check(res, { 'index 200': r => r.status === 200 })

  res = http.get(`${BASE_URL}/v1/products?page=1&size=10`, { headers })
  check(res, {
    'products 200': r => r.status === 200,
    'products business ok': r => r.json('code') === 200,
  })

  res = http.get(`${BASE_URL}/v1/products/1`, { headers })
  check(res, {
    'product detail ok': r => r.status === 200 && r.json('code') === 200,
  })

  const authHeaders = { ...headers, Authorization: `Bearer ${data.token}` }
  res = http.get(`${BASE_URL}/v1/cart`, { headers: authHeaders })
  check(res, { 'cart ok': r => r.status === 200 })

  sleep(Math.random() * 2 + 1)
}
```

## 助手 · 2026-04-27T11:49:27.239Z

压测时重点不是“把并发打上去”，而是控制风险、拿到可信数据。你这个项目是 Spring Boot + MySQL + Redis，注意这些点。

**环境隔离**

不要压生产环境。建议使用独立测试库、测试 Redis、测试账号。

压测前确认：

```text
MySQL、Redis、后端服务已启动
数据库有稳定的测试数据
测试账号可登录
不要和 Playwright、手工测试同时跑
```

你的后端 Hikari 连接池最大是 `20`，本地压到 50、100 VU 后，数据库连接池可能先成为瓶颈。

**先小后大**

不要一上来就 100、500 并发。推荐顺序：

```text
1 VU 冒烟
10 VU 基准
30/50 VU 阶梯
100 VU 峰值
30-50 VU 稳定性
```

如果冒烟都失败，后面的压测数据没有意义。

**别只看 HTTP 200**

这个项目很多接口会返回：

```json
{ "code": 200, "message": "success" }
```

所以 K6 里要同时检查：

```text
HTTP status 是否 200
业务 code 是否 200
响应体是否有 data
```

否则接口返回业务失败，但 HTTP 仍是 200，你会误判为成功。

**控制写接口**

读接口可以放心压：

```text
GET /api/
GET /api/v1/products
GET /api/v1/products/{id}
```

写接口要谨慎：

```text
POST /api/v1/auth/login
POST /api/v1/cart/items
POST /api/v1/orders/checkout
POST /api/v1/payments/{orderId}/pay
```

尤其是下单、支付、售后，会污染数据库。要压这些接口，必须准备清理脚本或一次性测试账号、测试商品、测试库存。

**设置停止标准**

出现这些情况应立即停止压测：

```text
http_req_failed > 5%
大量 500
大量 timeout
MySQL CPU 长时间过高
连接池耗尽
后端日志持续报错
机器明显卡死
```

压测不是把系统压垮才算完成，找到性能拐点就可以停。

**结果重点看**

K6 输出里重点关注：

```text
http_req_failed        失败率
http_req_duration p95  95% 响应时间
http_req_duration p99  99% 响应时间
checks                断言成功率
http_reqs             总请求数
iterations            完整业务迭代次数
```

建议目标：

```text
失败率 < 1%
checks > 99%
P95 < 800ms
P99 < 1500ms
无明显 5xx
```

**避免本机误差**

如果 K6 和后端都跑在同一台电脑上，压测结果会受本机 CPU、内存、磁盘影响。这个结果适合开发阶段找问题，不适合当正式容量结论。

更正式的做法是：

```text
一台机器跑 K6
一台机器跑后端
独立 MySQL/Redis
同时监控 CPU、内存、磁盘、网络、数据库连接数
```

**压测报告要记录**

每次压测至少记录：

```text
测试时间
测试环境
接口范围
并发模型
K6 命令
测试账号
P95/P99
失败率
主要错误
瓶颈判断
优化建议
```

最关键的一句话：先保证脚本真实模拟用户行为，再逐步加压；不要只追求高并发数字。

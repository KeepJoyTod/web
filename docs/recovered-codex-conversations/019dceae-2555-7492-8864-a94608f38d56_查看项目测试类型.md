## 助手 · 2026-04-27T11:25:41.000Z

UI 自动化测试就是：**用程序模拟真实用户在页面上的操作，自动验证页面功能是否正常**。

比如人工测试会这样做：

1. 打开首页
2. 点击“分类”
3. 输入搜索关键词
4. 点击商品
5. 加入购物车
6. 去登录页
7. 检查页面有没有跳转正确

UI 自动化测试会用脚本自动完成这些步骤，并判断结果是否符合预期。

在你这个项目里，UI 自动化测试主要就是 **Playwright 测试**，例如：

```text
frontend/tests/navigation.spec.ts
frontend/tests/login.spec.ts
frontend/tests/register.spec.ts
frontend/tests/product-detail.spec.ts
frontend/tests/cart.spec.ts
```

举个例子，`navigation.spec.ts` 做的就是 UI 自动化测试：

- 自动打开 `/`
- 自动点击底部导航
- 自动点击首页分类入口
- 自动搜索商品
- 自动点击商品卡片
- 自动验证是否跳转到正确页面

运行方式：

```powershell
cd d:\Java\class\projectKu\web\frontend
npm.cmd run test:navigation
```

UI 自动化测试和接口测试的区别是：

| 类型 | 测试对象 | 例子 |
| --- | --- | --- |
| 接口测试 | 后端 API | 请求 `/api/v1/products`，检查返回商品列表 |
| UI 自动化测试 | 页面和用户操作 | 打开首页，点击商品，检查是否进入详情页 |
| 压力测试 | 系统承载能力 | 模拟 1000 个用户同时访问 |
| 单元测试 | 某个函数或类 | 测试价格计算函数是否正确 |

简单说：

**UI 自动化测试 = 自动替代人工点页面，检查页面功能、跳转、按钮、表单、提示信息是否正常。**

它适合测：

- 登录注册流程
- 页面跳转
- 表单校验
- 商品详情
- 购物车操作
- 下单流程
- 未登录拦截
- 页面是否能正常加载

但它不太适合测：

- 高并发性能
- 数据库内部逻辑
- 单个后端方法的细节

所以在你这个项目里，UI 自动化测试已经有了，主要就是 `frontend/tests/*.spec.ts` 这些 Playwright 脚本。

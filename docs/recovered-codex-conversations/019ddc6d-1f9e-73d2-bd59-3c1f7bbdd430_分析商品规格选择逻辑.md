## 助手 · 2026-04-30T03:38:44.781Z

已继续完成只读分析，没有做任何代码修改。

**当前逻辑**
当前项目是“SKU 行驱动”的规格选择：后端商品详情返回 `product_skus`，每条 SKU 的 `attrs` 是一个 JSON 字符串；前端把第一条 SKU 的 `attrs` key 当作规格维度，然后用户逐项选择，最终用已选属性精确匹配一条 SKU。

关键链路是：

- 后端详情接口取商品、媒体、SKU，并把 `attrs` 从字符串解析成对象：[ProductServiceImpl.java](/D:/Java/class/projectKu/web/back/src/main/java/com/web/service/impl/ProductServiceImpl.java:47)
- 前端详情页如果后端没有 SKU，会生成 `规格=默认` 的默认 SKU：[ProductDetailView.vue](/D:/Java/class/projectKu/web/frontend/src/views/ProductDetailView.vue:693)
- 规格维度只取第一条 SKU 的 `attrs` keys：[ProductDetailView.vue](/D:/Java/class/projectKu/web/frontend/src/views/ProductDetailView.vue:339)
- 选中规格后，用所有维度完全匹配 `sku.attrs[k] === selected[k]` 找 `selectedSku`：[ProductDetailView.vue](/D:/Java/class/projectKu/web/frontend/src/views/ProductDetailView.vue:351)
- 价格、库存、按钮可用状态都跟 `selectedSku` 走：[ProductDetailView.vue](/D:/Java/class/projectKu/web/frontend/src/views/ProductDetailView.vue:371)
- 加入购物车保存的是 `sku.id` 和前端 SKU 价：[ProductDetailView.vue](/D:/Java/class/projectKu/web/frontend/src/views/ProductDetailView.vue:571)

**主要问题**
规格选择能跑，但模型比较脆。

1. 规格维度只看第一条 SKU。若同一商品后续 SKU 有额外维度或缺少维度，页面不会完整展示，甚至无法选到某些 SKU。

2. `attrs` 不是强结构，只是 `varchar(512)` 字符串：[init_db.sql](/D:/Java/class/projectKu/web/back/sql/init_db.sql:72)。我查了当前实库，`product_skus` 共 510 条，其中 25 条 `attrs` 不是合法 JSON，例如 `{"轴体":"樱桃银轴","RGB背光"}`。这类数据会让商品详情接口解析时报错，影响详情页加载。

3. 前端购物车用 SKU 价，但后端下单金额仍用商品基础价 `product.price`，没有用 `sku.price`：[OrderServiceImpl.java](/D:/Java/class/projectKu/web/back/src/main/java/com/web/service/impl/OrderServiceImpl.java:105)。这会导致“页面看到的金额”和“后端订单金额”不一致。

4. 下单时只按 `sku_id` 查 SKU，没有校验该 SKU 是否属于购物车里的 `product_id`：[OrderServiceImpl.java](/D:/Java/class/projectKu/web/back/src/main/java/com/web/service/impl/OrderServiceImpl.java:87)。如果传入错配 SKU，库存和价格归属可能串商品。

5. 库存存在双库存：`products.stock` 和 `product_skus.stock`。下单时有 SKU 也会同时扣 SKU 库存和 SPU 基础库存：[OrderServiceImpl.java](/D:/Java/class/projectKu/web/back/src/main/java/com/web/service/impl/OrderServiceImpl.java:101)。实库里大量商品 `products.stock` 不等于 SKU 库存之和，天然会漂移。

6. 购物车表没有 `(user_id, product_id, sku_id)` 唯一约束，只有 `idx_user_id`：[init_db.sql](/D:/Java/class/projectKu/web/back/sql/init_db.sql:133)。应用层尝试合并，但并发或查询条件问题可能产生重复项。

7. 列表页/收藏页有些“加购”直接使用 `skuId: 'default'`，绕过规格选择；对有 SKU 的商品来说，这会让后端按无 SKU 商品扣库存和计价，和详情页逻辑不一致。

**优化建议**
建议优先级这样排：

1. 先修数据：把非法 JSON 的 `attrs` 清理成合法键值，例如把 `"RGB背光"` 改成 `"背光":"RGB"` 这类结构。

2. 数据库把 `product_skus.attrs` 改成 MySQL `JSON` 类型，或至少加 `CHECK (JSON_VALID(attrs))`；同时加唯一约束，避免同一商品重复规格组合。

3. 后端下单改成以 SKU 为准：校验 `sku.product_id == cartItem.product_id`，订单价用 `sku.price`，库存用原子 SQL 扣减，例如 `stock >= quantity` 条件更新。

4. 明确库存口径：有 SKU 的商品，`products.stock` 只做冗余展示库存，定时或触发式由 SKU 求和；不要手工维护两套库存。

5. 前端规格维度不要只取第一条 SKU，应汇总所有 SKU 的 attrs keys，并对缺失维度做规范处理；更理想是后端直接返回 `specGroups` 和 `skuMatrix`。

6. 禁止有规格商品从列表页直接默认加购，改为跳详情页/规格弹窗。

结论：当前规格选择属于“轻量 SKU JSON 方案”，适合作业项目演示，但数据库约束和订单一致性还不够稳。最值得先处理的是非法 JSON、订单使用 SKU 价格、SKU 归属校验、库存扣减一致性这四个点。

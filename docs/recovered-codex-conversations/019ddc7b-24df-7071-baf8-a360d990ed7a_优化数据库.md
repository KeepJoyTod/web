## 助手 · 2026-04-30T04:02:15.089Z

可以简化，但不建议“为了少表而少表”。这个项目里最该保留多表的是订单、商品、库存这些核心链路；可以合并的是展示、权限、模拟支付这类外围功能。

我建议用“中度简化方案”。

**建议保留**
- `users`：用户、管理员角色都放这里。
- `categories`：商品分类。
- `products`：商品 SPU。
- `product_skus`：如果有规格库存，建议保留。
- `user_addresses`：如果用户能管理多个地址，保留。
- `cart_items`：购物车明细。
- `orders`：订单主表。
- `order_items`：订单商品快照，必须保留。
- `reviews`：评价。
- `aftersales`：售后流程独立，建议保留。
- `admin_operation_logs`：如果后台要审计操作，保留。

**可以合并或删除**
- `product_media` 可以合并到 `products`：
  - 加 `cover_url`
  - 加 `images JSON`
  - 适合你这种展示图不复杂的项目。

- `payments` 可以合并到 `orders`：
  - 在 `orders` 增加 `pay_status`、`pay_channel`、`trade_id`、`paid_at`
  - 当前项目是模拟支付、一个订单通常一条支付记录，单独建 `payments` 意义不大。
  - 如果以后要支持多次支付、退款、支付流水审计，再拆回 `payments`。

- `admin_permissions`、`admin_role_permissions` 可以删除：
  - 当前只有 `ADMIN` / `USER`，权限编码基本固定。
  - 后台权限可直接写在 Java 枚举或配置里。
  - 除非你要做“角色管理页面”，否则这两张表偏重。

- `coupons` 可按需求处理：
  - 如果只是下单减免，合并到 `orders`：`coupon_code`、`discount_amount`。
  - 如果要领券、过期、核销、库存，保留 `coupons`。

- `favorites` 可保留，也可以合成通用表：
  - 简化后表名可叫 `user_product_relations`
  - 字段：`user_id`、`product_id`、`relation_type`
  - `relation_type = FAVORITE`
  - 以后还能扩展浏览、点赞、想买等行为。

- `notifications` 可以视功能删除：
  - 如果前端没有站内信页面，先不要这张表。
  - 如果需要用户消息，保留即可。

**推荐简化后的表数量**
从现在约 18 张表，简化到 11 到 13 张比较合理：

```text
users
categories
products
product_skus
user_addresses
cart_items
orders
order_items
reviews
aftersales
admin_operation_logs
user_product_relations    可选
notifications             可选
```

**另一种方式：不减少物理表，用视图简化查询**
如果你的目标是“代码查询简单”，不一定要合表，可以建视图：

```text
v_product_card      商品卡片视图：商品 + 封面 + 分类 + 评分
v_order_admin       后台订单视图：订单 + 用户 + 支付信息
v_user_stats        用户统计视图：用户 + 订单数 + 售后数
```

这样数据库结构仍然清晰，Mapper 查询会简单很多。

我的建议：不要极限合表。优先合并 `product_media`、`payments`、`admin_permissions`、`admin_role_permissions`，其他表先保留。这样能明显降低复杂度，又不会破坏电商核心流程。

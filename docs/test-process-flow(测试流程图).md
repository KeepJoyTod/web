# 测试流程图

本文档用于说明项目从需求进入测试，到上线后观察的完整测试流程，适合作为团队测试执行总览。

## 1. 总体流程图

```mermaid
flowchart TD
    A[需求评审 / 范围确认] --> B[测试计划]
    B --> C[测试设计]
    C --> D[环境与数据准备]
    D --> E[冒烟测试]
    E --> F[功能测试]
    F --> G[接口 / 集成测试]
    G --> H[回归测试]
    H --> I[性能测试]
    I --> J{结果是否达标}
    J -->|否| K[分析报告与定位瓶颈]
    K --> L[开发 / 运维 / 数据库调优]
    L --> M[回归验证]
    M --> I
    J -->|是| N[缺陷收敛与测试结论]
    N --> O[验收测试 / UAT]
    O --> P[上线前检查]
    P --> Q[发布上线]
    Q --> R[上线后观察]
```

## 2. 压力测试后的处理闭环

压力测试结束后，通常不会直接结束测试，而是进入下面这个闭环：

```mermaid
flowchart LR
    A[执行压测] --> B[分析指标]
    B --> C[定位瓶颈]
    C --> D[系统调优]
    D --> E[回归验证]
    E --> F[复测]
    F --> G{是否达标}
    G -->|否| B
    G -->|是| H[输出最终报告]
```

重点关注指标：

- 成功率 / 错误率
- 响应时间 `p95`、`p99`
- 吞吐量 `QPS` / `TPS`
- CPU、内存、磁盘、网络
- 数据库连接数、慢 SQL、锁等待
- 应用线程池、缓存、消息队列、GC

## 3. 本项目建议落地顺序

结合当前仓库已有资产，建议测试执行顺序如下：

1. 先维护测试用例文档，例如 `docs/login-test-cases.md`、`docs/checkout-test-cases.md`。
2. 再执行 `frontend/tests/*.spec.ts` 中的自动化冒烟、功能和主流程测试。
3. 缺陷统一沉淀到 `docs/bug-report.md`。
4. 功能稳定后再执行 `k6` 性能测试，并将结果记录到 `docs/performance-test-records/`。
5. 当性能不达标时，回到“分析 -> 调优 -> 回归 -> 复测”闭环。

## 4. 阶段通过标准

建议按下面的判断逻辑推进：

- 冒烟不通过：停止后续测试，先修阻塞问题。
- 功能测试不通过：不进入性能测试。
- 主流程回归不稳定：不输出验收结论。
- 压测未达标：不直接判定可上线，必须复测。
- 验收未通过：不进入正式发布。
